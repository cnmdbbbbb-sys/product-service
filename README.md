# product-service
1️⃣ 项目全流程详解（我们做了什么）
这个项目遵循了现代云原生企业最标准的 “代码即基础设施” 和 “声明式运维” 规范，整体流程分为四大核心阶段：

阶段一：本地多节点集群搭建（基础设施层）
操作：利用 Windows WSL2 (Ubuntu) 算力，使用 Kind (Kubernetes in Docker) 在本地拉起了一个包含 1 个 Control-plane（控制平面）和 2 个 Worker（工作节点）的 K8s 物理多节点集群。

意义：完全模拟了生产环境的分布式架构，摆脱了单节点（Minikube）的玩具属性。

阶段二：云端安全镜像中转站构建（云端存储层）
操作：在 AWS（亚马逊云） 平台开通了 ECR (Elastic Container Registry) 私有镜像仓库。为了遵循企业安全“最小权限原则”，没有使用 Root 账号，而是专门创建了一个只有 ECR 读写权限的 IAM 用户，并签发了 API 密钥。

阶段三：CI（持续集成）自动化流水线搭建（自动化层）
操作：将微服务代码托管至 GitHub，在代码库根目录下编写了 .github/workflows/deploy.yml 声明式流水线文件。利用 GitHub Actions 的云端免费算力，在代码触发 push 时自动执行：
拉取代码 ➡️ 注入 AWS 加密凭证 ➡️ 自动安全登录 AWS ECR ➡️ 容器化打包（Dockerfile） ➡️ 分别打上 latest 和 Git Commit SHA 双标签 ➡️ 推送上云。

阶段四：CD（持续部署）GitOps 闭环（收官层）
操作：在本地 Kind 集群中部署了 ArgoCD。利用 AWS CLI 获取 12 小时免密 Token 并制作成 K8s 内部的 imagePullSecrets。最后编写 ArgoCD Application 清单，让 ArgoCD 实时监听 GitHub 仓库。

2️⃣ 核心实现功能（实现了什么指标）
通过这套架构，你最终实现了以下四大生产级技术指标：

0 成本全自动流水线（CI/CD 自动化）：研发人员只需在网页端或本地执行 git push 修改一行代码，后续的打包、编译、跨云传输、集群部署全部由机器自动完成，实现 Code-to-Cloud 闭环。

端到端安全授信（Security & IAM）：本地不保存任何 AWS 根凭证，GitHub 通过加密的 Repository Secrets 传递暗号，K8s 集群通过定期刷新的 docker-registry 密钥与 AWS ECR 进行私密通信，保障企业资产不外泄。

GitOps 强一致性（声明式驱动）：GitHub 仓库成为了系统状态的“唯一事实来源（Single Source of Truth）”。集群的实际运行状态由 ArgoCD 自动向 Git 仓库对齐，消除了传统运维“手动敲命令配置集群”导致的配置漂移。

零停机平滑升级（Rolling Update & High Availability）：利用 K8s 的滚动更新策略结合 readinessProbe（就绪检查），新旧 Pod 在升级过程中平滑交棒，新兵不通过健康检查，老兵绝不下线，实现了 0 毫秒服务中断。

3️⃣ 对实际生产的帮助与价值（为什么企业需要它）
在面试中，面试官不仅看你“怎么做”，更看重你是否理解“为什么这么做”。这套架构在实际生产中能为企业解决以下痛点：

📉 降本增效（ROI 最大化）
传统生产痛点：企业为了构建流水线，往往需要购买昂贵的 Jenkins 服务器、专用的高防内网自建仓库（如 Harbor），并花费大量人力维护。

本项目的价值：采用 “混合云算力借用” 思想。把耗费 CPU 算力的“镜像打包”丢给 GitHub 免费构建机；把高带宽、高可用的“存储”丢给 AWS ECR 免费额度；本地只保留最轻量的 K8s 运行业务。帮初创企业实现了 0 成本的工业级 CI/CD 演练。

🛡️ 规避“删库跑路”与环境混乱（环境一致性）
传统生产痛点：测试环境一个样，生产环境另一个样。因为某个运维在服务器上偷偷改了一个配置，导致项目上线直接崩溃。

本项目的价值：一切皆代码（Everything as Code）。无论是基础设施、CI 流程还是 K8s 部署清单，全部记录在 Git 历史中。任何变动可审计、可追溯、可一键回滚。

🚀 极大地释放研发生产力（提升交付频次）
传统生产痛点：以前部署新版本，研发要打包发给运维，运维挑人少的半夜凌晨加班部署，提心吊胆怕出故障。

本项目的价值：得益于自动滚动更新与健康检查机制，研发在白天大流量时期也可以随时触发流水线。K8s 自动灰度替换，出问题自动拦截，真正实现了“随时想发就发，版本无缝切换”。
